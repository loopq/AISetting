# Feishu PRD Skill

## 工作流程

1. **收集输入** — 确认用户已提供 commits、需求描述、代码变更或设计稿
2. **获取模板结构** — 使用 `feishu-mcp` fetch-doc 获取飞书模板：`https://stickerstyle.feishu.cn/docx/UNsmdZiPIoezPIxcT7TcYgUznwx`
3. **生成 PRD** — 按下方规范生成内容，按需取舍模块
4. **询问写入飞书** — 生成完毕后询问用户是否调用 `feishu-mcp` 创建/更新文档

## 写作核心原则

- **用户视角**：描述「用户看到什么、感受到什么」，不是「系统做了什么」
- **禁止堆砌技术术语**：非技术读者能看懂
- **数据引用**：commit 中有性能数据直接引用；没有则给出定性描述（如「显著缩短」「基本消除」）
- 每个 commit/变更点对应功能描述中的一段

## 文档结构（按需取舍，删除不涉及的模块）

| 模块 | 说明 |
|------|------|
| **需求背景** | 用户当前遇到什么问题？为什么要做？ |
| **需求目标** | 一句话概括期望效果 |
| **功能描述** | 逐条描述用户可感知的变化 |
| **验收标准** | 可量化、可直接拿来验收的清单 |
| **性能对比** | 优化前后数据表格（如适用） |
| **影响范围** | 哪些页面/流程受影响？兼容性风险？ |
| **排期** | 基于 commit 时间给出实际完成日期 |

删除不涉及的模块（如原型图、多语言买点、UI 规范等）。

## 输出格式

- 飞书文档风格的 Markdown
- 表格使用标准 Markdown 表格语法
- 层级清晰，H2/H3 标题

## 写入飞书

如用户确认写入飞书：
- 创建新文档：`feishu-mcp create-doc`
- 更新已有文档：`feishu-mcp update-doc`（用户提供文档链接）

详细写作示例参见 `references/prd-examples.md`
